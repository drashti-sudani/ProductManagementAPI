# ProductManagementAPI

High-level documentation and quick-start for the ProductManagementAPI (.NET 8) reference implementation.

## Overview

- Purpose: RESTful API for managing Products (CRUD) with authentication, validation, logging and tests.
- Framework: .NET 8 (ASP.NET Core Web API)
- Layers: API, Application, Domain, Infrastructure

Tech highlights
- EF Core (SQL Server) for data access
- JWT access + refresh token authentication
- FluentValidation for request validation
- Serilog for structured logging
- Swagger for API documentation
- Docker + Docker Compose for containerized development

## Swagger

- The project exposes Swagger UI in Development mode. Start the app and navigate to:
  - https://localhost:5001/swagger or http://localhost:5000/swagger
- Security: Swagger is configured with a Bearer token security definition. Click "Authorize" and paste `<access-token>`.
- To improve the autogenerated schema enable XML comments in the API project (.csproj):

  <GenerateDocumentationFile>true</GenerateDocumentationFile>

  Then add the generated XML to SwaggerGen in Program.cs using `c.IncludeXmlComments(xmlPath)`.

## Code documentation

- C# server code: use XML doc comments (triple-slash ///) on public controllers, services and DTOs. Example:

  /// <summary>
  /// Creates a new product.
  /// </summary>
  public Task<ProductDto> Create(ProductCreateDto request) { ... }

- Frontend / JS client: use JSDoc for functions you plan to publish or share. Example:

  /**
   * Login and return tokens.
   * @param {{username:string,password:string}} creds
   * @returns {{accessToken:string,refreshToken:string}}
   */
  async function login(creds) { }

## Authentication (High level)

1. Client POSTs credentials to `/api/v1/auth/login`.
2. Server validates credentials and issues:
   - short-lived JWT access token (used in Authorization: Bearer <token>)
   - longer-lived refresh token (stored server-side and returned to client)
3. Client calls protected endpoints with the access token.
4. When access token expires, client POSTs refresh token to `/api/v1/auth/refresh` to obtain new access token (and optionally new refresh token).
5. Logout endpoint revokes refresh token server-side.

Security notes:
- Use HTTPS in production.
- Keep access token lifetime short and rotate refresh tokens.
- Store refresh tokens safely (prefer server-side DB); protect against CSRF/XSS when storing tokens in browsers.

## Environment setup (High level)

Prerequisites
- .NET 8 SDK
- SQL Server (local or Docker)
- Docker & Docker Compose (optional)
- Visual Studio 2022/2026 or VS Code

Quickstart (local, using Docker Compose)

1. Copy or update configuration values (secrets) before running.
   - Environment variables used by the app include:
	 - `ConnectionStrings__DefaultConnection`
	 - `JwtSettings__Secret`
	 - `ASPNETCORE_ENVIRONMENT`
2. Start containers: `docker compose up --build`
3. The API attempts to apply EF Core migrations on startup (see Program.cs) so database schema is created/updated automatically.
4. Open Swagger to exercise endpoints.

Running locally without Docker

1. Configure appsettings.Development.json or user secrets with connection string and JWT secret.
2. Run DB (SQL Server) locally and ensure the connection string points to it.
3. Run: `dotnet run --project src/API`

Database migrations (manual)

- To add a migration: `dotnet ef migrations add <Name> -p src/Infrastructure -s src/API`
- To update DB: `dotnet ef database update -p src/Infrastructure -s src/API`

Note: the project currently runs migrations at startup; for production prefer running migrations from CI/CD or an admin job.

## Deployment (High level)

Containerized (recommended)
- Build image: `docker build -f src/API/Dockerfile -t product-api:1.0 .`
- Push to registry (GHCR, Docker Hub, ACR) and deploy to your target (VM, Kubernetes, Azure App Service).
- Provide secrets via environment variables or secrets manager (Key Vault, Secrets Manager, Kubernetes Secrets).

CI/CD checklist
- Build and run unit tests
- Produce and push Docker image
- Run DB migration step (pipeline task or admin job)
- Deploy to staging and run integration tests
- Promote to production

Non-containerized production
- Host app on Windows/Linux servers or PaaS (App Service) and connect to managed SQL Server. Manage migrations and secrets carefully.

## Notes about Windows Integrated Authentication

- Linux SQL Server container images do NOT support Windows Integrated Authentication. If you need Integrated Security for local DB on Windows, run the API on the host (not inside Linux containers) or use Windows containers with gMSA (advanced).
